# AgentM Core Abstraction Inventory

This document separates fixed SDK/core contracts from implementation backlog.
The refactor target is that future work implements Protocol backends and policy
atoms without reopening trajectory, store, or cancellation modeling.

## Boundary Rule

Do not add optional backends directly into minimal SDK core unless the backend
is required for embedded SDK sessions to run.

Core defines stable ports, invariants, service keys, and default local
behavior. Optional packages provide durable storage, remote execution,
gateway/presenter integrations, and deployment-specific policy.

## Recovery Boundaries

| Path | Behavior | Reason |
| --- | --- | --- |
| Missing trajectory head with committed turns | Rebuild the node projection from the authoritative `TrajectoryStore` turn log, then retry explicit-head lookup. If the rebuilt projection still has no active head, fail fast. | Node projection is a repairable read model; append order must not fall back to leaf inference. |
| Node projection append/rebuild failure after turn commit | Commit resource and effect participants, preserve the already committed turn log, stop the current driver, and repair the projection on resume/next append. | The turn log and committed external world are authoritative. A broken read projection must not leave the effect participant prepared but uncommitted. |
| Authoritative append failure after resource apply/effect prepare | Restore the effect world first, then abandon the resource transaction and its journal; collect both cleanup failures without running them concurrently. | Both participants may touch the same workspace. Concurrent rollback creates a filesystem race and can turn a determinate append failure into an indeterminate world state. |
| Effect snapshot contains SDK control-plane state | Backend composition is invalid. Snapshot implementations must exclude all independently durable SDK stores; the local reference backend excludes `.agentm` by default. | Resume runs each store's recovery before restoring the world. Restoring a captured resource journal or trajectory log would rewind a commit that already reached a known outcome. |
| Node-chain projection without node store/head | Fail fast when a `NodeChainContextProjection` is installed but no `TrajectoryNodeStore` or active head can supply its chain. | Exact replay must not silently widen back to turn replay. |
| Compaction summary/state mismatch | Write the immutable summary `ResourceRef` before saving `ContentReplacementState`. Missing summary content, invalid coverage, or an upstream projection that changes the uncompacted turn suffix fails fast; no implicit tail-only fallback is used. | Compaction replaces model-visible history, so a partial or silently widened recovery would change semantics. Orphaned immutable summaries are harmless and can be collected separately. |
| Environment restore failure on resume | Raise `EnvironmentRestoreError` unless a registered `EnvironmentRestoreFailureHandler` successfully activates degraded read-only enforcement. | External world state is part of trajectory correctness; continuing after failed restore requires an enforceable host action, not a passive policy flag. |
| Provider selection before first commit | An explicit `ProviderResolver` may choose the initial provider while no history exists. Once history commits, persisted `ProviderSessionIdentity` pins provider/model/active-set identity; resume never guesses a missing identity from messages, model names, or provider registration order. | Provider selection is policy before history; after history, silent model drift corrupts replay semantics. |
| Provider request raises after retries | Commit a `ProviderRequestFailed` turn, including the provider exception type, detail, and partial-event count, before failing the trigger receipt. The turn and its node messages remain queryable but are marked non-replayable. | A session header without the failed request is not an auditable trajectory. Replaying an uncertain partial request into a later provider call would also be incorrect. |
| Process exits with an incomplete turn | Retain the latest `TurnCheckpoint` in the selected trajectory store and expose it through trajectory queries. Resume, fork prefixes, provider context, compaction, and node projection ignore it because it has no committed `Outcome`. A later run starts from the committed prefix; hosts may retain or explicitly discard the orphan checkpoint as diagnostic evidence. | Materialized messages must survive a crash, but replaying a tool call or partial provider step would duplicate uncertain effects. Explicit checkpoint state preserves both requirements without fabricating a committed termination cause. |

## Store Backend Contract

Every configured trajectory backend exposes one `TrajectoryStorage` pair even
when its physical storage differs. The authoritative turn port and rebuildable
node port share one backend selection, lifecycle, and namespace.

Required node identity and ordering fields:

- `id`
- `session_id`
- `root_session_id`
- `parent_session_id`
- `seq`
- `parent_id`
- `logical_parent_id`
- `branch_id`
- `head_id`

Required turn/message join fields:

- `turn_id`
- `turn_index`
- `round_index`
- `message_index`
- `kind`
- `role`
- `visibility`
- `timestamp`

Required routing/query fields:

- `agent_id`
- `is_sidechain`
- `tool_call_id`
- `tool_name`
- `cache_key`
- `content_ref`

The `tool_call_id` and `tool_name` fields are logical inverted indexes. A
message node may contain multiple tool calls/results; SQL stores may normalize
them into a child table, while JSONL stores may scan arrays.

Head storage is separate from node storage. A backend must support a unique
current head per `(session_id, head_id)` and efficient branch/agent/sidechain
selection by `(root_session_id, session_id, branch_id, agent_id, is_sidechain)`.
