# 2026-05-21 — Prefix-replay follow-up fixes

Three integration-time defects surfaced after commit `5cff9c6` (prefix-replay
landing). Each is independently small; sequencing matters only because Fix 1
touches a Protocol in `core.abi`.

## Fix 1 — sidecar filename must match the persisted main session file

**Where**: `core/abi/extension.py` (Protocol), `core/runtime/session_helpers.py`
(`SessionView` already-provided implementation), `adapters/agentm.py` (the
two sites that compose `replay_log_path` and `ReplayRecord.root_session_id`).

**Why**: `api.root_session_id` is the OTel trace_id (uuid generated by the
substrate at session_factory.py:119); the persisted JSONL session file is
named after `SessionManager.header.id` (generated independently by
`JsonlSessionManager.new_session`). These differ on every real run, so
`agent-from-reminder` can't relocate the source session.

**Change shape**:
1. Add `get_session_id(self) -> str` to `ReadonlySession` Protocol.
2. Implement on `SessionView` by delegating to `self._sm.get_session_id()`
   (the underlying `SessionManager` already has it).
3. In `adapters/agentm.py`, prefer `api.session.get_session_id()` when it
   returns a non-empty string; fall back to `api.root_session_id` (covers
   in-memory / unpersisted sessions whose header is `None`).
4. Keep `ReplayRecord.root_session_id` field name unchanged — only the value
   written into it changes.

**Boundary risk**: `ReadonlySession` is a `runtime_checkable` Protocol with
exactly one impl in-tree (`SessionView`). Adding a method is backward-
compatible for callers, but any third-party impl would need to add the new
method — flagged in the design doc.

**Tests**:
- Existing `test_prefix_replay_cli.py` already uses
  `source_mgr.get_session_id()` for the sidecar name; that test stays green.
- Existing `test_reminder_injector.py` uses MagicMock for `api.session`, so a
  new attribute call is auto-mocked — no edit needed.
- New small test: build a real `SessionManager`-backed `SessionView`, route
  through a helper that picks the id the adapter would write, and assert it
  equals the `SessionManager.get_session_id()` (the same id used in the
  persisted filename).

## Fix 2 — bump default audit drain timeout to 600s

**Where**: `adapters/agentm.py` (`_DEFAULT_SHUTDOWN_TIMEOUT_S` constant +
the matching `config_schema` description).

**Why**: 60s is too short for Warpgate-fronted endpoints (~14s/call). Bump
to 600s; document the knob.

**Tests**: none — knob, not fail-stop.

## Fix 3 — `llmharness-replay` bridges env → provider config

**Where**: `replay/cli.py` (`_parse_provider` callers and downstream
`_PHASE_RUNNERS`).

**Why**: `agentm` CLI bridges `AGENTM_PROVIDER`, `AGENTM_MODEL`,
`OPENAI_BASE_URL`, etc. into the provider config dict through
`ProviderRegistry.build()`. The replay CLI just forwards `--provider`
verbatim, so users with custom endpoints hit 307s / SSL errors.

**Change shape**: when `--provider` is omitted, build the default provider
spec via `DEFAULT_PROVIDER_REGISTRY.build(provider, {"model": model})`
where `provider` and `model` come from `AGENTM_PROVIDER` / `AGENTM_MODEL`
(or registry defaults). When `--provider <bare-module>` (no `:{}`) is given,
treat the module as the provider id, look it up in the registry, and apply
the same env-bridge. When `--provider 'module:{json}'` is given (explicit
config), don't touch — explicit wins.

Reuse `DEFAULT_PROVIDER_REGISTRY.build()` directly. No new abstraction.

**Tests**: one unit test that sets `AGENTM_PROVIDER=openai` +
`OPENAI_BASE_URL=https://example.com` in a monkeypatched env, calls the new
resolver helper, asserts the config dict contains `base_url`.
