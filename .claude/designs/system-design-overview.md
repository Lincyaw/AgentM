# Design: AgentM System Architecture

**Status**: DRAFT
**Last Updated**: 2026-03-08

---

## Overview

AgentM is a **hypothesis-driven multi-agent orchestration framework** built on LangGraph. A central Orchestrator (Supervisor node) coordinates multiple Sub-Agents (Subgraph nodes) to perform root cause analysis (RCA) and other diagnostic tasks.

### Design Goals

| Goal | Description |
|------|-------------|
| Hypothesis-driven reasoning | All reasoning happens in the Orchestrator; Sub-Agents only collect data |
| Config-driven system | Prompts, tools, models configured via YAML files |
| Agent isolation | Sub-Agents have private state; communicate via shared `messages` field |
| Orchestrator supervision | Monitor Sub-Agent execution, interrupt, redirect, inject instructions |
| Trajectory recording | Hierarchical execution traces with hypothesis evolution, RL export |
| Failure recovery | Resume from checkpoint, debug replay from any point |
| Multi-system support | Generic SDK wrapper supports different diagnostic patterns |
| Visual debugging | Frontend dashboard for topology, execution monitoring, conversation history |

## Theoretical Foundation: Hypothetico-Deductive Method

The Orchestrator is modeled as a **state machine** following the **Hypothetico-Deductive Method** from experimental science:

```
OBSERVE → HYPOTHESIZE → EXPERIMENT → ANALYZE → CONCLUDE
   ↑                                              │
   └──── REFUTED (new cycle) ←────────────────────┘
```

### Four-Phase Diagnostic Flow

| Phase | Scientific Step | Actor | Description |
|-------|----------------|-------|-------------|
| **1. Exploration** | Observe | Sub-Agents (parallel) | Collect raw data from multiple sources |
| **2. Generation** | Hypothesize | Orchestrator (LLM) | Generate candidate hypotheses from observations |
| **3. Verification** | Experiment + Analyze | Sub-Agents → Orchestrator | Investigate data, then interpret results |
| **4. Confirmation** | Conclude | Orchestrator | Confirm root cause or refute and loop back |

Key design choices:
- **No confidence scores** — LLM-generated confidence is unreliable; use three-value verdict (`confirmed / rejected / partial`)
- **No prediction anchoring** — Orchestrator does NOT set expected outcomes before experiments, avoiding confirmation bias
- **Adversarial review** (feature-gated) — A Devil's Advocate Sub-Agent challenges conclusions to counter LLM confirmation bias
- **Async task dispatch** — Orchestrator submits multiple concurrent tasks via LangGraph `Send` API, like a real team leader

Sub-Agent returns a **three-block result** per verification:

| Block | Content |
|-------|---------|
| **Investigation Data** | Raw investigation results (metrics, logs, states) |
| **Reasoning** | Supporting reasons, rejecting reasons, neutral observations |
| **Verdict** | `confirmed` / `rejected` / `partial` |

> Detail: See [orchestrator.md](orchestrator.md) for Notebook structure, state machine formalization, feature gates, and phase implementation.

## State Management (Mode 2)

AgentM uses **minimal messages + Notebook** architecture:

- `messages` field keeps only 2-3 messages (initial + current interaction)
- `DiagnosticNotebook` is the primary working memory
- Frontend reconstructs the full conversation from Notebook for display
- LLM input is the structured Notebook, not a long message list

```python
class ExecutorState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # Minimal
    notebook: DiagnosticNotebook                          # Working memory
    task_id: str
    current_phase: str
```

This is fully supported by LangGraph: custom dataclass fields alongside messages, partial state updates, and `RemoveMessage` API.

### Context Compression

Even with Mode 2, long-running RCA tasks can exhaust context windows. Two-layer compression addresses this:

- **Sub-Agent layer**: When tool call count exceeds threshold, compress message history into a structured 8-section summary via `pre_model_hook`. Full history remains in checkpoints.
- **Orchestrator layer**: On phase transition, compress completed phase's `exploration_history` into a `PhaseSummary`. Active phase retains full detail.

Compression is a **prompt optimization only** — checkpoints retain full uncompressed data for trajectory export, replay, and observability. Each compression creates a `CompressionRef` in checkpoint metadata recording the compressed range (`from_id`, `to_id`), enabling O(1) drill-down.

**History Recall**: After compression, Agents can call a `recall_history` tool to query their pre-compression checkpoint messages on demand — retrieving raw metric data, tool call parameters, and intermediate reasoning that the summary omitted. This shares the same conceptual interface as cross-task knowledge retrieval (natural language query → relevant data).

> Detail: See [orchestrator.md](orchestrator.md) for compression data structures, checkpoint interaction, recall tool design, and configuration.

## Configuration System

Three-layer configuration, all YAML-based:

### Layer 1: System Config (`system.yaml`)

```yaml
system:
  type: "hypothesis_driven"  # or "sequential", "decision_tree"
  models:
    gpt-4: { api_key: "${OPENAI_API_KEY}" }
  storage:
    checkpointer: { backend: "postgresql", url: "${DB_URL}" }
```

### Layer 2: Orchestrator Config (`orchestrator.yaml`)

```yaml
orchestrator:
  model: "gpt-4"
  temperature: 0.3
  prompts:
    system: "templates/orchestrator_system.txt"
    hypothesis_generation: "templates/generate_hypotheses.txt"
  tools: [dispatch_task, interrupt_agent, inject_instruction]
  monitoring:
    stream_mode: ["updates", "custom"]
    max_execution_time: 3600
```

### Layer 3: Agent Configs (`agents/*.yaml`)

```yaml
agent:
  name: "infrastructure"
  model: "gpt-4o-mini"
  temperature: 0
  prompt:
    template: "templates/agents/infrastructure_system.txt"
  tools: [check_cpu, check_memory, check_disk]
  state_schema: "infrastructure_private_state"
  shared_fields: ["messages"]
```

### Layer 4: Runtime Config (context_schema)

```python
@dataclass
class ExecutorContext:
    user_id: str
    task_id: str
    language: str = "en"

result = graph.invoke(input, config, context=ExecutorContext(...))
```

Switching scenarios requires **only config + prompt files**, zero code changes.

## Monitoring & Intervention

Orchestrator acts as Team Leader with full visibility:

- **Stream monitoring**: `stream(subgraphs=True)` captures all Sub-Agent events in real-time
- **Interrupt**: Pause Sub-Agent execution at checkpoints
- **Instruction injection**: `update_state()` to inject new instructions into running Sub-Agent
- **Redirect**: Reroute task to a different Sub-Agent

> Detail: See [orchestrator.md](orchestrator.md) for intervention implementation.

## Trajectory System

### Hierarchical Trace Capture

```
Main Trace (Orchestrator)
├── Phase 1: Exploration
│   ├── Sub-Agent: Infrastructure (child trace)
│   ├── Sub-Agent: Logs (child trace)
│   └── Sub-Agent: Database (child trace)
├── Phase 2: Hypothesis Generation
│   └── H1, H2, H3 created
├── Phase 3: Verification
│   ├── H1 verification → confirmed
│   └── H2 verification → rejected
└── Phase 4: Confirmation
```

Two complementary approaches:
- **Post-hoc**: `get_state_history()` to walk checkpoint chain
- **Real-time**: `stream(subgraphs=True)` during execution

### RL Training Export

```python
@dataclass
class RLTransition:
    state: dict          # Orchestrator state at step
    action: dict         # Task dispatched
    reward: float        # Success/failure signal
    next_state: dict     # State after action
    agent: str           # Which agent executed
```

Export as JSONL — one transition per line.

## Failure Recovery & Debug Replay

```python
# Resume from failure
result = graph.invoke(None, config)  # None = resume from last checkpoint

# Fix state and retry
graph.update_state(config, {"messages": [...]})
result = graph.invoke(None, config)

# Fork from a past checkpoint for debugging
history = list(graph.get_state_history(config))
fork_config = graph.update_state(history[3].config, {"messages": [...]})
debug_result = graph.invoke(None, fork_config)
```

Error handling strategy by type:
- Rate limit → exponential backoff retry
- Timeout → simple retry
- Validation error → fix state and retry
- Fatal error → rollback to last good checkpoint

## Multi-System Support (SDK Wrapper)

Different agent system types share the same core framework (BaseOrchestrator, PhaseManager, StateSchemaFactory), differentiated by configuration:

```python
# Hypothesis-driven RCA
orchestrator = AgentSystemBuilder.build(system_type="hypothesis_driven", config=...)

# Sequential diagnosis (only config changes)
orchestrator = AgentSystemBuilder.build(system_type="sequential", config=...)

# Memory Extraction — cross-task knowledge building (same framework, different state + phases)
memory_agent = AgentSystemBuilder.build(system_type="memory_extraction", config=...)
```

The **Memory Extraction** system reuses the same Supervisor + Subgraph architecture to process completed RCA trajectories and extract failure patterns, diagnostic skills, and system knowledge into a persistent Knowledge Store (LangGraph Store). The RCA Orchestrator can then query this knowledge via `search_knowledge` tool during hypothesis generation.

## Key Design Decisions

| Decision | Rationale | Trade-off |
|----------|-----------|-----------|
| Hypothetico-Deductive state machine | Formal scientific method foundation | Must handle cycle-back on refutation |
| Supervisor + Subgraph | Leverages LangGraph's native multi-agent capabilities | Graph must be compiled at startup |
| Minimal messages + Notebook | Avoids context window explosion; structured working memory | Frontend must reconstruct conversation |
| No confidence scores | LLM-generated confidence is unreliable; three-value verdict | Less granular ranking |
| No prediction anchoring | Avoids confirmation bias in LLM | Verdict relies on post-hoc analysis |
| Adversarial review (feature-gated) | Devil's Advocate counters LLM confirmation bias | Extra LLM call per verification |
| Async task dispatch | Orchestrator submits concurrent tasks; time-multiplexing | Results aggregation complexity |
| Feature gates | All behaviors config-toggleable | Config surface area grows |
| Config-driven everything | Zero code changes for new scenarios | Config validation needed |
| Checkpoint-based trajectory | Automatic capture, no extra overhead | Must reconstruct tree from linear chain |
| PostgreSQL persistence | Production-grade reliability | External service dependency |
