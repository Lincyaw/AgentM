# Design: Extension-as-Scenario (Unified Pluggability Model)

**Status**: CURRENT
**Created**: 2026-04-30
**Supersedes**: the standalone `Scenario` abstraction in `generic-state-wrapper.md` (kept as historical context; the protocol described here replaces the wiring layer).
**Builds on**: [pluggable-architecture.md](pluggable-architecture.md)

---

## 1. Core Insight

**A scenario is a *composition* of atomic extensions, expressed as data — not code.** Extensions are the atoms; scenarios are recipes that say "load these atoms with this configuration".

| Property | Decision |
|---|---|
| What is an extension? | A single-responsibility module exporting `install(api, config)`. The unit of registration. |
| What is a "scenario"? | A configuration file (YAML / dict) listing which extensions to load and with what configs. **Pure data, no code.** |
| Built-in vs. user scenarios? | Same mechanism. Both are just lists-of-tuples that `AgentSession.create` consumes. |
| What lives in core? | Only mechanism (kernel + Operations ports). All policy and behavior lives in extensions. |
| Sub-agents, plan mode, RCA, general-purpose? | Each = a scenario YAML composing atoms. None are core features, none are bespoke modules. |
| LLM provider? | An extension that registers a `StreamFn`. |

**Why this is the right shape**:
- *No privileged path*: a third-party scenario is structurally identical to a built-in scenario — both are YAML files referencing extension module paths.
- *Composability*: the same atom (`tool_read`, `permission`, `system_prompt`) is reusable across scenarios. No duplicated implementations.
- *Diffable, shareable, AI-generatable*: scenarios are text. They can live in PRs, be generated by another agent, and version-controlled like any config.

### 1.1 Atom vs. scenario, concretely

```yaml
# src/agentm/extensions/scenarios/general_purpose.yaml — a scenario is data
extensions:
  - module: agentm.extensions.builtin.tool_read
  - module: agentm.extensions.builtin.tool_bash
  - module: agentm.extensions.builtin.tool_edit
  - module: agentm.extensions.builtin.tool_write
  - module: agentm.extensions.builtin.system_prompt
    config:
      prompt: "You are a helpful coding assistant. ..."
```

```yaml
# src/agentm/extensions/scenarios/rca.yaml — different recipe, same atoms
extensions:
  - module: agentm.extensions.builtin.tool_read
  - module: agentm.extensions.builtin.tool_hypothesis_store
  - module: agentm.extensions.builtin.system_prompt
    config:
      prompt: "You are an RCA analyst. Data only — never conclude."
  - module: agentm.extensions.builtin.permission
    config:
      deny: ["bash", "edit", "write"]    # data-only constraint
```

The user-facing call:

```python
session = AgentSession.create(
    extensions=load_scenario("rca") + [("my_org.custom_ext", {})],
    provider=("agentm.extensions.builtin.llm_anthropic", {"model": "claude-opus-4-7"}),
)
```

`load_scenario(name) -> list[tuple[str, dict]]` is a ~30-line YAML reader. **It is sugar, not new mechanism.** The kernel only ever sees the flattened `extensions=[...]` list.

## 2. Extension Model

### 2.1 What an Extension Is

An extension is a callable:

```python
ExtensionFactory = Callable[[ExtensionAPI, dict[str, Any]], None | Awaitable[None]]
```

It receives the API + a config dict, performs registrations imperatively, and returns. That is the entire surface. There is no class hierarchy, no `setup() → Wiring` indirection, no special metadata file.

```python
def install(api: ExtensionAPI, config: dict[str, Any]) -> None:
    api.register_tool(my_tool)
    api.on("tool_call", my_guard)
    api.on("before_agent_start", inject_my_system_prompt)
    api.register_command("/mything", handler)
```

Every extension is structurally identical. RCA, permission gates, the Anthropic provider, and a one-line user hack all use this same shape.

### 2.2 Lifecycle

```
session_create → for each configured extension: call install(api, config)
              → AgentSession ready
              → events fire as the loop runs
              → session_shutdown: emit event; extensions clean up via on("session_shutdown")
```

Extensions are stateless from the harness's view. If they need state, they capture it in closures or attach it via `api.events` (cross-extension EventBus).

### 2.3 Loading

```python
# atoms only — fully explicit
session = AgentSession.create(
    extensions=[
        ("agentm.extensions.builtin.permission", {"deny": ["bash"]}),
        ("agentm.extensions.builtin.dedup", {}),
        ("agentm.extensions.builtin.tool_read", {"file_ops": LocalFileOperations()}),
        ("agentm.extensions.builtin.sub_agent", {"max_workers": 4}),
        ("my_org.custom_extension", {"foo": "bar"}),
    ],
    provider=("agentm.extensions.builtin.llm_anthropic", {"model": "claude-opus-4-7"}),
)

# OR: scenario recipe + extra atoms — the common case
session = AgentSession.create(
    extensions=load_scenario("rca") + [
        ("agentm.extensions.builtin.sub_agent", {"max_workers": 4}),
    ],
    provider=("agentm.extensions.builtin.llm_anthropic", {"model": "claude-opus-4-7"}),
)
```

Path is a Python module path resolved via `importlib`. The module must export an `install` callable. No metadata files, no plugin manifest — convention is the manifest. Python-native (entry points are deferred to v0.2 if needed).

## 3. ExtensionAPI (v0 surface)

Keep this small. Every method earns its keep — we pick the minimal subset that covers the 8 acceptance scenarios in `pluggable-architecture.md` §6.

```python
class ExtensionAPI(Protocol):
    # --- Event subscription (typed overloads per channel) ---
    def on(self, channel: str, handler: Handler) -> Unsubscribe: ...
    def add_observer(self, callback: ObserverCallback) -> Unsubscribe: ...

    # --- Registrations ---
    def register_tool(self, tool: Tool) -> None: ...
    def register_command(self, name: str, spec: CommandSpec) -> None: ...
    def register_provider(self, name: str, config: ProviderConfig) -> None: ...
    def register_message_renderer(self, custom_type: str, renderer: Renderer) -> None: ...

    # --- Actions ---
    def send_user_message(self, content: str | list[Content]) -> None: ...
    async def spawn_child_session(self, config: AgentSessionConfig | dict[str, Any]) -> Any: ...
    def set_service(self, name: str, obj: Any) -> None: ...
    def get_service(self, name: str) -> Any | None: ...
    def get_resource_writer(self) -> ResourceWriter: ...

    # --- Read-only context (lazy properties; cwd/model/session may change) ---
    @property
    def cwd(self) -> str: ...
    @property
    def session(self) -> ReadonlySession: ...
    @property
    def model(self) -> Model | None: ...
    @property
    def events(self) -> EventBus: ...   # for cross-extension communication
```

`add_observer` is the typed surface for passive event-bus observation; atoms
must not patch `EventBus` internals. `spawn_child_session` accepts the legacy
dataclass or a kwargs mapping so atoms do not import harness internals.
`set_service` / `get_service` provide a per-session registry for atom-owned
state that should not live in module globals.

**Deferred (add when first extension needs them)**: shortcuts/keybindings, UI primitives (select/confirm/input), flag registration, theme management, status/footer/widget injection. These are TUI concerns; we'll add them when the interactive mode lands.

## 4. AgentSession (orchestrator)

The fat-but-thin façade per `pluggable-architecture.md` §4. Holds references; runs no business logic.

```python
@dataclass
class AgentSessionConfig:
    cwd: str
    extensions: list[tuple[str, dict]]      # (module_path, config)
    provider: tuple[str, dict]               # provider extension + config
    initial_messages: list[AgentMessage] = field(default_factory=list)
    session_dir: str | None = None           # None = in-memory only
    resource_loader: ResourceLoader | None = None  # None = default FS loader

class AgentSession:
    @classmethod
    def create(cls, **config) -> "AgentSession": ...

    # The single user-facing entrypoint
    async def prompt(
        self,
        text: str,
        *,
        images: list[ImageContent] | None = None,
        signal: asyncio.Event | None = None,
    ) -> list[AgentMessage]: ...

    # Public properties exposed to extensions and modes
    @property
    def bus(self) -> EventBus: ...
    @property
    def session_manager(self) -> SessionManager: ...
    @property
    def resources(self) -> ResourceLoader: ...
    @property
    def tools(self) -> list[Tool]: ...     # registered tools, filtered by allowlists
    @property
    def model(self) -> Model | None: ...

    # Lifecycle
    async def shutdown(self) -> None: ...
```

`prompt()` does what `pluggable-architecture.md` §4 lists, in <100 lines. No branch carries policy; extensions own policy.

## 5. SessionManager (entry tree)

```python
@dataclass(frozen=True, slots=True)
class SessionEntry:
    type: str             # "message" | "compaction" | "branch_summary" | custom
    id: str               # uuid7
    parent_id: str | None
    timestamp: float
    payload: Any          # type-specific; opaque to core

class SessionManager(Protocol):
    def append(self, entry: SessionEntry) -> None: ...
    def get_active_branch(self) -> list[SessionEntry]: ...   # leaf → root via parent_id
    def get_messages(self) -> list[AgentMessage]: ...        # active branch, message entries only
    def fork_at(self, entry_id: str) -> "SessionManager": ...  # new manager, same store
    def navigate_to(self, leaf_id: str) -> None: ...
    def find(self, entry_id: str) -> SessionEntry | None: ...

class JsonlSessionManager(SessionManager): ...      # default, ~/.agentm/sessions/<id>.jsonl
class InMemorySessionManager(SessionManager): ...   # for tests / embedding
```

Single JSONL file, append-only. `payload: Any` lets extensions persist their own structured data (RCA hypothesis index, trajectory analysis snapshots, etc.) without forking the format.

## 6. ResourceLoader (project context)

```python
@dataclass(frozen=True, slots=True)
class Skill:
    name: str
    description: str
    body: str
    source: str          # file path or descriptor

@dataclass(frozen=True, slots=True)
class PromptTemplate:
    name: str
    body: str
    source: str

@dataclass(frozen=True, slots=True)
class ContextFile:
    body: str
    source: str          # AGENTS.md / CLAUDE.md path

class ResourceLoader(Protocol):
    def get_skills(self) -> list[Skill]: ...
    def get_prompt_templates(self) -> list[PromptTemplate]: ...
    def get_context_files(self) -> list[ContextFile]: ...
    def reload(self) -> None: ...
```

Default impl walks `~/.agentm/{skills,prompts}/` and `cwd → parent dirs` for `AGENTS.md`. Embedded SDK callers pass an in-memory or DB-backed loader.

## 7. Catalog: atomic extensions + scenario recipes

### 7.1 Atomic extensions (single-responsibility code modules)

Each is one Python module with `install(api, config)` doing **one thing**. The atom is what gets reused across scenarios.

#### Tool atoms

| Module | Registers | Notes |
|---|---|---|
| `extensions.builtin.tool_read` | `read` | Delegates to `FileOperations` from config / `api.get_operations().file` |
| `extensions.builtin.tool_bash` | `bash` | Delegates to `BashOperations` from config |
| `extensions.builtin.tool_edit` | `edit` | Delegates exclusively to `ResourceWriter` |
| `extensions.builtin.tool_write` | `write` | Delegates exclusively to `ResourceWriter` |
| `extensions.builtin.tool_hypothesis_store` | `add_hypothesis`, `update_hypothesis`, `list_hypotheses` | Owns an in-memory store; persists via `api.session.append_entry("hypothesis", …)` |
| `extensions.builtin.tool_trajectory_loader` | `load_trajectory`, `summarize_trajectory`, `find_event`, `compare_trajectories` | Reads JSONL trajectory files |
| `extensions.builtin.tool_submit_plan` | `submit_plan` | Appends a `plan` SessionEntry; emits `plan_submitted` event |

#### Prompt / context atoms

| Module | Behavior |
|---|---|
| `extensions.builtin.system_prompt` | `before_agent_start`: prepends `config["prompt"]` to the assembled system prompt. Single most-reused atom. |
| `extensions.builtin.turn_reminder` | (was `system_reminder`) Re-injects `config["reminder"]` every `every_n_turns`. |
| `extensions.builtin.micro_compact` | Compacts context near limit; emits `before_compact` / `after_compact`. |
| `extensions.builtin.llm_compaction` | Performs LLM-driven compaction when the durable session branch exceeds the reserved-token threshold; rebuilds context from the session tree after persisting a `compaction` entry. |
| `extensions.builtin.trajectory` | Records every event to JSONL. |

#### Policy atoms

| Module | Behavior |
|---|---|
| `extensions.builtin.permission` | `on("tool_call")` returns `{"block": True}` for tools matching `config["deny"]` (deny wins) or not in `config["allow"]`. |
| `extensions.builtin.tool_filter` | At `agent_start`, removes tools whose names appear in `config["deny"]`. |
| `extensions.builtin.dedup` | Skips recently-repeated tool calls. |
| `extensions.builtin.cost_budget` | Tracks token cost; emits `cost_budget_exceeded` event. |
| `extensions.builtin.tool_result_budget` | Truncates oversized tool outputs. |
| `extensions.builtin.file_mutation_queue` | Wraps `edit`/`write` with per-path `asyncio.Lock`. Must load AFTER tool atoms (§10b.4). |

#### Multi-agent atom

| Module | Behavior |
|---|---|
| `extensions.builtin.sub_agent` | Registers `dispatch_agent`, `check_tasks`, `inject_instruction`, `abort_task`; manages child `AgentSession`s. |

### 7.2 Scenario recipes (data files, not code)

Live under `src/agentm/extensions/scenarios/<name>.yaml`. Each lists the atoms to load and their configs. **No `install()` function — these are pure data.**

| Scenario YAML | Composes |
|---|---|
| `general_purpose.yaml` | `tool_read` + `tool_bash` + `tool_edit` + `tool_write` + `system_prompt`(generic coding) |
| `rca.yaml` | `tool_read` + `tool_hypothesis_store` + `system_prompt`(RCA framing) + `permission`(deny mutating tools) |
| `trajectory_analysis.yaml` | `tool_read` + `tool_trajectory_loader` + `system_prompt`(analyst framing) |
| `plan_mode.yaml` | `tool_submit_plan` + `system_prompt`(plan-mode framing) + `permission`(deny mutations) |

Loading: `load_scenario(name)` reads the YAML, returns `list[tuple[module_path, config]]`. Caller concatenates with their own extensions.

### 7.3 LLM providers (separate top-level package, also extension-shaped)

| Module | Purpose |
|---|---|
| `llm.anthropic` | Native Anthropic `StreamFn` registered via `register_provider` |
| `llm.openai` | (deferred, on-demand) |
| `llm.langchain_bridge` | (deferred, on-demand) Wraps `langchain-core` `BaseChatModel` as a `StreamFn` |

LLM providers (separate top-level package, also extension-shaped):

| Module | Purpose |
|---|---|
| `llm.anthropic` | Native Anthropic `StreamFn` registered via `register_provider` |
| `llm.openai` | (later) |

## 9. Implementation Plan & Agent Dispatch

### Phase 1 — v2 Harness Foundation (single agent, sequential)

**Why sequential**: the four modules are tightly coupled. ExtensionAPI is created by AgentSession; AgentSession holds SessionManager and ResourceLoader. Splitting across agents would race on shared types.

**Deliverable**:
- `harness/extension.py` (ExtensionAPI Protocol + impl + loader)
- `harness/session.py` (AgentSession)
- `harness/session_manager.py` (entry tree + InMemory + Jsonl impls)
- `harness/resource_loader.py` (Protocol + DefaultResourceLoader)
- Tests in `tests/unit/harness_v2/` covering: extension lifecycle, session.prompt() end-to-end with mock StreamFn, JSONL round-trip, resource discovery on a temp dir

### Phase 1b — First LLM Provider (parallel with Phase 1)

**Why parallel**: depends only on `core/kernel/`, no harness dependency.

**Deliverable**:
- `llm/anthropic.py` — native `StreamFn` for Anthropic Messages API
- Tests in `tests/unit/llm/anthropic/` using cassette-style HTTP recording (or mocked transport)

### Phase 2.0a — Operations ports (sequential prerequisite for Group D1)

**Group A0**: define `FileOperations` / `BashOperations` Protocols + `LocalFileOperations` / `LocalBashOperations` defaults in `src/agentm/core/operations.py`. Group D1 depends on this.

### Phase 2 — Atomic extensions + scenario recipes (parallel batch)

Each atom is independent (only depends on `core/kernel/`, `core/operations.py`, `harness/extension.py`). Spawn implementer agents in parallel, one per logical group:

- **Group A (policy atoms)**: permission, tool_filter, dedup, cost_budget, tool_result_budget
- **Group B (context atoms)**: turn_reminder, system_prompt, file_mutation_queue, micro_compact, trajectory
- **Group C (multi-agent atom)**: sub_agent
- **Group D1 (tool atoms)** — depends on A0: tool_read, tool_bash, tool_edit, tool_write, tool_hypothesis_store, tool_trajectory_loader, tool_submit_plan
- **Group D2 (scenario recipes + loader)** — depends on A, B, D1: 4 YAML files (general_purpose, rca, trajectory_analysis, plan_mode) + `extensions/loader.py` (`load_scenario`)

### Phase 2.5 — Legacy deletion (single sweep)

After all groups land and the cross-extension composition test passes, run the deletion sweep listed in §8 in one PR. No staged retirement — the new tree is the only tree.

## 10. Open Questions

1. **`@tool` decorator targeting**: existing `core/tool.py @tool` decorator produces langchain-flavored tools. It should emit `kernel.Tool` directly. Schedule for Phase 2 (decorator refactor doesn't block extension work — extensions can use `FunctionTool` or define classes directly).
2. **Config format**: keep YAML for now (matches existing), but parse to `(module_path, config)` tuples. Schema validation deferred.
3. **Provider-as-extension vs provider-as-config-field**: design currently has provider as a separate field on `AgentSessionConfig`. Could collapse into `extensions=[...]` if we want maximum uniformity. Decision: keep separate field for clarity; provider is *uniquely required*, extensions are *uniquely additive*. Acceptable asymmetry.
4. **Session-shutdown extension cleanup**: do we need an explicit `on_unload` per extension or rely on `on("session_shutdown")`? Decision: rely on event for v0; promote to first-class hook only if a real use case requires it.
5. **Async vs sync extension factories**: pi supports both. Python: same — `install` may be sync or async; loader awaits if coroutine.

## 10b. Phase 2 Pre-Design Decisions (from Phase 1 review)

Phase 1 + 1b are landed and reviewed. Before launching Phase 2 in parallel, these decisions must be locked because they affect multiple extensions.

**Status (2026-04-30)**: §10b.1, .2, .3, .4, .5, .7, .8, .9, .10 contracts and the supporting kernel/harness wiring landed as **Phase 2.0b** (see `tasks/2026-04-30-phase2-group-0b.md`). Concretely: the missing event dataclasses are declared in `harness/events.py`; `ReadonlySession.append_entry(type, payload, parent_id=None) -> str` is implemented; `AgentSession.create` now subscribes to `cost_budget_exceeded` and emits `session_ready`; `prompt()` drains `_pending_user_messages` before each turn and was extracted into helpers to stay a <100-line dispatcher. §10b.6 (Operations ports) is delivered by Phase 2 Group A0; §10b.11 below is the new `session_ready` contract added during 2.0b.

### 10b.1 New events to add (non-breaking; pure additions)

| Event | Why | Owners |
|---|---|---|
| `before_compact` / `after_compact` | compaction extension needs cancel/replace semantics | `micro_compact`, future `agent_memory` |
| `before_send_to_llm` (with token-counted payload) | `cost_budget` needs hook before the bytes hit the wire (`context` event fires too early — no usage data yet) | `cost_budget` |
| `child_session_start` / `child_session_end` | sub-agent lifecycle visibility for trajectory + cost rollups | `sub_agent`, `trajectory` |
| `render_message` (custom_type → component) | message-renderer extensions need a place to push UI; without it, TUI mode has no contract | deferred until interactive mode |

`before_tool_call` distinct from `tool_call`: rejected. The current `tool_call` event already supports both mutate-in-place and `{"block": True}` returns; a second event would force extensions to pick one channel arbitrarily.

### 10b.2 The `context`-vs-SessionManager divergence

When a `context` handler rewrites the message list (e.g. `micro_compact`), the rewrite is what the **kernel** sends to the LLM but is **not** persisted to SessionManager. This is intentional: SessionManager is the truth (full history), `context` is a per-turn transformation layer. Compaction extensions that want to *also* persist the compacted form must explicitly `session_manager.append(compaction_entry(...))`.

Document this contract here so Phase 2 extension authors don't trip over it. It is **not** a bug.

### 10b.3 Payload serialization contract (for SessionManager)

`SessionEntry.payload` must be one of: a dataclass (will be walked recursively), a dict, a list, or a JSON scalar. Custom objects with non-serializable fields will be silently stringified by `default=str`. Phase 2 extensions putting structured `details` in compaction/branch entries (e.g. RCA hypothesis tree) must use dataclasses.

### 10b.4 Slash-command runner

`api.register_command` records commands but nothing dispatches them yet. Phase 2 must add a small command-runner: when `prompt(text)` sees `text.startswith("/")`, look up the command and invoke its handler instead of going through the agent loop. Single ~30-line addition to `AgentSession.prompt`. Schedule this with whichever Phase 2 group lands first.

### 10b.5 Sub-agent nesting & state

`AgentSession` holds per-instance state (active provider closure box, EventBus). Nested `AgentSession.create` for sub-agents will get fresh state automatically — no globals to fight. The `sub_agent` extension is responsible for managing child session lifecycle and propagating events upward via the parent bus if desired.

The `sub_agent` extension's `inject_instruction` v0 semantics: append the new message to a per-child queue; the queue is drained when the current `prompt(...)` resolves (i.e. the next user-turn starts with the queued message). True streaming-mid-turn injection is deferred.

Provider re-use: a child session is constructed with `provider=parent.provider` (same `StreamFn` callable). The Anthropic `StreamFn` is implemented as a closure that captures only the API key and model defaults — it has **no per-session state** and is safe to share. Group C tests must include an explicit assertion that two concurrent child sessions sharing one StreamFn complete cleanly.

### 10b.6 FileOperations / BashOperations ports

To honour `pluggable-architecture.md` §3.2 (acceptance scenario 2 — bash over SSH), the `general_purpose` scenario's `read`/`bash`/`edit`/`write` tools must NOT call `subprocess`/`pathlib` directly. Instead:

```python
# src/agentm/core/abi/operations.py  — Protocols (atom + harness import)
class FileOperations(Protocol):
    async def read_file(self, path: str) -> bytes: ...
    async def access(self, path: str) -> bool: ...
    async def is_dir(self, path: str) -> bool: ...
    async def list_dir(self, path: str) -> list[str]: ...

class BashOperations(Protocol):
    async def exec(self, cmd: str, *, cwd: str, timeout: float | None = None,
                   env: dict[str, str] | None = None,
                   on_data: Callable[[bytes], None] | None = None,
                   signal: asyncio.Event | None = None) -> ExecResult: ...

# src/agentm/extensions/builtin/operations_local.py — default impls (atom)
# install(api, config) calls api.register_operations(file=..., bash=...)
class LocalFileOperations: ...     # stdlib-backed read environment
class LocalBashOperations: ...     # asyncio.subprocess-backed
```

Tool atoms use a deliberate hybrid seam: read-only file tools (`tool_read` /
`tool_grep` / `tool_find` / `tool_ls`) accept `config["file_ops"]` and
default to `api.get_operations().file`; write tools (`tool_write` / `tool_edit`)
use `api.get_resource_writer()` exclusively so managed-resource versioning and
constitution-path rejection remain the single mutation chokepoint. Swapping read
transport = pass a different `FileOperations`; redirecting or auditing writes =
replace `ResourceWriter`.

This is delivered by **Phase 2 Group A0** before Group D1 so the tool atoms can be built against the ports from day one.

### 10b.7 ReadonlySession.append_entry expansion

Multiple Phase 2 extensions (`micro_compact`, `rca`, `trajectory_analysis`) need to write structured `SessionEntry` records back into the session tree. The current `ReadonlySession` only exposes `get_messages()`. Decision: **expand `ReadonlySession`** with one method:

```python
def append_entry(self, type: str, payload: Any, parent_id: str | None = None) -> str:
    """Append a custom entry to the active branch. Returns the new entry id."""
```

It returns the new entry id (compaction needs the id to build parent_id chains). Implementation delegates to the underlying `SessionManager.append`. Done in Phase 2 Group B alongside `micro_compact` (smallest blast radius — one method on one Protocol).

### 10b.8 cost_budget overflow mechanism

`cost_budget` raises a custom event `cost_budget_exceeded` (carrying `{used: float, limit: float, currency: str}`) when accumulated cost crosses the limit. The agent loop subscribes (in `AgentSession.prompt`) to this channel and, on first emission, flips an internal flag that causes the next iteration to terminate with `agent_end(stop_reason="budget")`. Pure event-bus-based — no exceptions thrown across handler boundaries. Document the channel name and payload in `harness/events.py`.

### 10b.9 plan_mode as a scenario recipe (not an atom)

Plan mode is **not** a single extension. It is a scenario YAML (`scenarios/plan_mode.yaml`) composing three atoms:

```yaml
extensions:
  - module: agentm.extensions.builtin.tool_submit_plan
  - module: agentm.extensions.builtin.system_prompt
    config:
      prompt: "You are in plan mode. Investigate; do not mutate state. Call submit_plan when ready."
  - module: agentm.extensions.builtin.permission
    config:
      deny: ["bash", "edit", "write"]
```

The atom `tool_submit_plan`:
- Registers `submit_plan(plan: str)`.
- `execute` calls `api.session.append_entry("plan", {"text": plan})` and emits `plan_submitted` on `api.events`.
- Returns `ToolResult(content=[TextContent("plan submitted")], extras={"plan_submitted": True})`.

The "deny mutations" behavior is just `permission` with the right `deny` list — no special-case code.

### 10b.10 Test directory convention

All Phase 2 tests live under `tests/unit/extensions/builtin/<name>/test_*.py`. The earlier plan text saying `tests/unit/extensions/<name>/` is superseded.

### 10b.11 `session_ready` event

`AgentSession.create` emits `session_ready` (payload: `SessionReadyEvent` carrying `cwd`, `session_id`, `tool_names`, `command_names`, `model`) **after** every extension is loaded and the active provider is picked, but **before** the first `prompt` runs. This is the only timing point where every extension is guaranteed to see the *final* tool list, command set, and active model.

Use cases:
- `tool_filter` (Group A): scrub tool list by name after every other extension has registered its tools — cannot be done in `install()` because tool registrations from later extensions are not yet visible there.
- `trajectory` (Group B): capture the initial inventory snapshot.
- Any "post-install audit" extension (cost-budget reset, session-startup banner, etc.).

Contract:
- Emitted exactly once per session.
- Synchronous w.r.t. `create` — handlers are awaited before `create` returns.
- Tool list passed in the payload is a tuple snapshot at emit time; mutations after this point (e.g. tools registered from a `session_ready` handler itself) are visible to subsequent `prompt` calls but do not retroactively appear in the snapshot.

### 10b.12 `append_entry` lives on ReadonlySession, not ExtensionAPI

The §10b.7 method is reachable as `api.session.append_entry(...)`, **not** `api.append_entry(...)`. The previous draft of ExtensionAPI carried an `append_entry(custom_type, payload)` no-op stub that was never drained — it has been removed in 2.0b. Extension authors persisting structured payloads (compaction summaries, hypothesis snapshots, plan submissions) must go through `ReadonlySession`.

### 10b.13 `send_user_message` queue is drained at the top of every `prompt`

`api.send_user_message(content)` queues a payload onto an internal FIFO; `AgentSession.prompt` drains the queue before appending the caller's text, so queued items act as turn-prefix context. Used by `sub_agent.inject_instruction` and any extension that wants to nudge the conversation without driving a synchronous turn. The queue is **per-session** — child sessions have their own.

## 11. Single-File, Self-Contained Extension Contract

**Why this matters**: future work has agents editing their own extensions. Keeping the surface tight and *mechanically checkable* shrinks the agent's exploration space and prevents whole categories of breakage (cross-file refactors, hidden dependencies, undocumented config keys).

### 11.1 Hard rules

1. **One file, one extension.** A new built-in extension is a single Python file at `src/agentm/extensions/builtin/<name>.py`. **No subpackages**, no helper modules. If the file would exceed ~300 LoC, the responsibility is too broad — split it into two extensions instead. Opt-in contrib packages may group multiple single-file atoms under `contrib/extensions/<package>/` when mounted explicitly with `--extension`; package-internal helper modules are allowed only for shared install-time parsing/rendering, and each public atom file still exports its own `MANIFEST` + `install`.
2. **Module-level `MANIFEST`.** Every built-in extension exports a module-level constant `MANIFEST: ExtensionManifest` (frozen dataclass; defined in `agentm.extensions`). This is the *only* way an extension declares what it is. No magic comments, no plugin entry-point files.
3. **`install(api, config)` is the only callable surface.** Internal helpers may exist as module-private functions (`_underscore_prefixed`); they are never imported by other modules.
4. **Allowed imports** (enforced by validator):
   - stdlib
   - `agentm.core.abi.*` — types and Protocols (Tool, Message, StreamFn, events, FileOperations / BashOperations / ExecResult, SkillRecord, PromptTemplateRecord, Operations, CompactionSettings, ...)
   - `agentm.core.lib.*` — pure-function utility shelf (edit_diff, frontmatter, path_utils, text_truncate); use as stdlib
   - `agentm.extensions` (the public surface — `ExtensionManifest`)
   - **Forbidden**: `agentm.core._internal.*` (constitution-private; reach via `api.get_operations()` / `api.skills` / `api.prompt_templates` / `api.catalog` / `api.compaction`), `agentm.core.runtime.*` (extensions never reach inside the runtime substrate), other `agentm.extensions.builtin.*` modules (no atom-to-atom coupling).
5. **Config schema is declared, not implied.** `MANIFEST.config_schema` is a JSON-Schema `dict` (or `None` for "no config accepted"). The validator uses it to (a) reject unknown keys at load time, (b) provide an explicit contract for agent-driven config generation.

### 11.2 `ExtensionManifest` shape

```python
@dataclass(frozen=True, slots=True)
class ExtensionManifest:
    name: str                              # unique within builtin/; matches filename stem
    description: str                       # one-line summary
    registers: tuple[str, ...]             # e.g. ("tool:read",), ("event:tool_call",), ("provider:anthropic",)
    config_schema: dict[str, Any] | None   # JSON-Schema for the config dict; None ⇒ no config
    requires: tuple[str, ...] = ()         # other extension names that MUST be loaded earlier
    conflicts: tuple[str, ...] = ()        # other extension names that MUST NOT be loaded together
```

`registers` is a tag list, not a free-form string: tags follow `<kind>:<id>` where `<kind>` ∈ `{tool, event, command, provider, renderer}`. The validator parses these and uses them to detect ordering/conflict issues.

Scenario/session loading topologically sorts manifest-bearing extensions by `requires` before installation and fails fast when a declared dependency is absent. Built-in atoms with no peer dependency still spell `requires=()` with an inline rationale so the absence is auditable.

### 11.3 Auto-discovery

`agentm.extensions.discover.discover_builtin() -> dict[str, BuiltinEntry]` walks `extensions/builtin/`, imports each module, and returns a name → entry map. Used by:
- `load_scenario(name)` — validates that every module path in a scenario YAML resolves to a discovered atom, and surfaces a clear error otherwise (no more `ModuleNotFoundError` wrapped in `ExtensionLoadError`).
- The validator (§11.4).
- Future tooling: an agent listing "what atoms exist" reads this map; nothing else.

Discovery is **memoized per process** so production loads pay the directory walk once. Flat-file contrib discovery intentionally scans only `contrib/extensions/*.py`; privileged contrib packages such as `contrib.extensions.cc` are loaded only by explicit module path and are not auto-discovered from their package directory.

### 11.4 Validator

`agentm.extensions.validate.validate_builtin() -> list[ValidationIssue]` runs the following checks against every discovered module:

1. Module is a single file under `builtin/`, not a package directory.
2. Defines a callable `install` with signature `(api, config)`.
3. Defines `MANIFEST: ExtensionManifest`.
4. `MANIFEST.name` matches the module stem (`permission` ↔ `permission.py`).
5. Imports are within the allow-list above.
6. Every tag in `MANIFEST.registers` parses as `<kind>:<id>` with a known kind.
7. `MANIFEST.requires` and `MANIFEST.conflicts` reference names that exist (or are documented forward references for not-yet-landed atoms).
8. If `config_schema` is set, it is a syntactically valid JSON-Schema dict.
9. Peer atom names referenced as string literals in code must appear in `MANIFEST.requires`.
10. AST hygiene rejects private ExtensionAPI reflection, ExtensionAPI attribute
   overwrites, mutable module-level dict/list/set globals without `Final`,
   f-string dynamic imports under `agentm.*`, and `isinstance` downcasts to
   concrete harness service classes. Atoms must consume harness capabilities
   through ExtensionAPI and Protocol methods.

Test integration: `tests/unit/extensions/test_extension_contract.py` calls `validate_builtin()` and fails the suite on any issue. **This is the gate every new extension PR must pass mechanically — an agent self-editing an extension knows it broke the contract before any human reads the diff.**

### 11.5 What goes in a single-file extension PR

Adding a new atom = exactly one new file at `src/agentm/extensions/builtin/<name>.py`. The author does **not** edit:

- `agentm/extensions/loader.py` (auto-discovery)
- `agentm/extensions/__init__.py` (public surface stays fixed)
- Any harness file
- Any other extension

The PR may add **one** test file under `tests/unit/extensions/builtin/<name>/test_*.py` (path locked, §10b.10) plus optionally edit a scenario YAML to opt the new atom in. That is the entire allowable diff surface.

### 11.6 Why not a plugin manifest file (TOML/YAML)?

Considered and rejected: a separate manifest file doubles the source-of-truth and forces validators to cross-check two artifacts. Inline `MANIFEST = ExtensionManifest(...)` keeps everything an agent needs to reason about in one place — read the file, you know the contract; edit the file, the diff carries the manifest change.
